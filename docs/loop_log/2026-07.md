## Phase 1 错题录入与认知诊断真实闭环（2026-07-18，规则引擎四色风险 + 前置深度溯源 + 三视角评审 P0/P1 修复）#book_structure #deployment

### 触发问题

Phase 0 仅奠定魔搭入口与四层架构基线，错题录入 Tab 和认知诊断 Tab 都是规则引擎兜底（按错题数 TOP3 找薄弱点），无法演示真实诊断链路。Phase 1 需要让「录入 → 诊断」链路可演示且有单元测试守护，同时清掉 Phase 0 评审 P1 工程债。LLM 接入留到 Phase 1.5（待 DASHSCOPE_API_KEY）。

### 根因

1. **认知诊断缺四色风险等级与前置溯源**：Phase 0 规则引擎只按错题数 TOP3 列薄弱点，无风险等级区分（red/yellow/green/gray）、无错因聚合、无前置依赖溯源，教师无法判断"哪些是直接薄弱、哪些是前置牵连"，也无法得到"先补前置再补薄弱"的补漏路径。
2. **SQLite 并发与隔离性不足**：默认 journal_mode 无 WAL，多用户并发录入可能 database is locked；DB_PATH 在模块加载时绑定默认路径，单元测试无法用 tmp_path 注入独立 SQLite，测试间共享 `/workspace/data/wrongbook.db` 导致数据互相污染。
3. **Python 默认参数陷阱**：`def insert_wrong_problem(..., db_path: Path = DB_PATH)` 在函数定义时捕获 DB_PATH 值，monkeypatch `store.DB_PATH` 后默认参数不更新，导致 isolated_db fixture 注入的路径未被使用，20 个测试失败。
4. **HaloRead 沿袭迁移不彻底**：`bug-reporting.md` 和 `dev-checklist.md` 仍写 `--output output`（knowtrace 用 content/）；`DEEP_READING_MOCK` 环境变量在 conftest.py 已改 `LLM_MOCK` 但通用工程 skill（dev-selfcheck/verification-before-completion/git-merge-guardian）仍引用 `--output output`；3 个通用工程 skill 文档契约与 knowtrace 实际路径分裂。
5. **api.py 跨层调用 + 图谱路径硬编码两处**：`api.py` 的 `get_wrongbook_list` 直接调 `store.list_wrong_problems` 绕过 Service 层；`api.py` 和 `diagnosis.py` 各自独立计算 `data/knowledge_graph/七年级上册.json` 路径，Phase 1.5 切 LLM 时会再加第三份重复。
6. **graph_path fixture 图谱隔离失效**：api 测试的 `graph_path` fixture 只 patch `api.KNOWLEDLEDGE_GRAPH_PATH`，但 `handle_diagnosis_run → diagnose_student` 实际读 `diagnosis.GRAPH_PATH`，导致诊断链路落到真实图谱文件，测试给出虚假信心。
7. **gray 评级定义但永不产出**：`_evaluate_risk` 定义了 gray（未做题，0 错题），但 `diagnose_student` 只遍历 `kp_count`（有错题的知识点），count 永远 ≥ 1，gray 是死代码，UI 承诺的"⚪ gray 未做题"列永远不出现。
8. **MD_BACKUP_DIR 模块级常量是默认参数陷阱的变体**：`MD_BACKUP_DIR = DB_PATH.parent / "wrongbook_md"` 在 import 时绑定 DB_PATH，monkeypatch `store.DB_PATH` 后 `MD_BACKUP_DIR` 不跟着变，MD 备份仍写到生产目录。

### 修复

1. **认知诊断升级（规则引擎版）**：四色风险等级（red ≥4 道 / yellow 2-3 道 / green ≤1 道 / gray 0 道）+ 错因聚合（按知识点 ID 聚合 error_type 列表，去重保序）+ 前置深度溯源（递归查 2 层 prerequisites，去重保序）+ 推荐补漏路径（先补前置，再补薄弱）。DiagnosisResult schema 新增 node_risks 字段，Phase 1.5 切 Qwen3-Max 时 schema 不变。
2. **SQLite 并发与隔离性**：启用 WAL 模式 + 30s busy timeout；DB_PATH 支持 `KNOWTRACE_DB_PATH` 环境变量注入；所有 store 函数默认参数改为 `db_path: Path | None = None`，函数体内 `if db_path is None: db_path = DB_PATH` 读取模块级变量，避免默认参数陷阱。
3. **文件名 sanitize 升级**：student_id 和 knowledge_point_id 都用 `re.sub(r"[^\w\-]", "_", value)` 安全处理；`_sanitize_filename(None)` 返回 "unknown" 防御 None 入参。
4. **MD_BACKUP_DIR 陷阱修复**：新增 `_get_md_backup_dir()` 函数动态读取 `store.DB_PATH`，`_write_md_backup` 内部用它代替模块级常量；保留 `MD_BACKUP_DIR` 模块级常量供测试 rglob 查找用。
5. **gray 评级真实产出**：`diagnose_student` 改为遍历 `set(nodes_by_id.keys()) | set(kp_count.keys())` 取并集评级，图谱中未做题的节点也产出 gray 评级。
6. **_load_graph JSON 异常兜底**：加 `try/except (json.JSONDecodeError, OSError)` 降级为空图谱，不阻断诊断。
7. **isolated_db fixture 扩展**：patch `exam.OUTPUT_DIR` 和 `voice.VOICE_OUTPUT_DIR` 到 tmp_path，避免 exam/voice 测试向真实 `data/exam_output`/`data/voice_output` 写文件污染工作区。
8. **graph_path fixture 同时 patch 两个模块**：api 测试的 `graph_path` fixture 同时 patch `api.KNOWLEDLEDGE_GRAPH_PATH` 和 `diagnosis.GRAPH_PATH`，确保诊断链路也走临时图谱。
9. **3 个通用工程 skill 路径对齐**：`dev-selfcheck/SKILL.md`、`verification-before-completion/SKILL.md`、`git-merge-guardian/SKILL.md` 中 `--output output --strict` 改为 `--output content --strict`，`output/<专栏>` 改为 `content/<专栏>`。
10. **.env.example 登记 KNOWTRACE_DB_PATH**：新增 `# KNOWTRACE_DB_PATH=` 注释行，让新开发者知晓此变量存在。
11. **新增 3 个守护测试**：`test_default_db_path_resolved_at_call_time`（守护默认参数陷阱，禁止改回 `db_path=DB_PATH`）、`test_handle_diagnosis_run_yellow_risk_shows_yellow_emoji`（api 层 yellow 评级覆盖）、`test_diagnose_gray_risk_for_unattempted_kp`（守护 gray 评级真实产出）。强化 `test_handle_diagnosis_run_with_red_risk` 断言 root_causes + recommendation_path，守护图谱遍历在 api 层的集成。`test_wal_mode_enabled` 改为强断言 wal + tmpfs 降级时显式 skip。

### 关键教训

1. **Python 默认参数陷阱有多种变体，修复要彻底**：函数默认参数 `db_path: Path = DB_PATH` 是经典陷阱（定义时绑定）；模块级常量 `MD_BACKUP_DIR = DB_PATH.parent / "wrongbook_md"` 是同陷阱的变体（import 时绑定）。修复模式：函数默认参数改 `None` + 函数体内读模块级变量；模块级常量改为函数内动态读取。判断标准：任何在 import 时绑定全局变量的写法，monkeypatch 该全局变量后都不会生效，必须改为调用时读取。
2. **graph_path fixture 需同时 patch 调用链上所有模块的路径常量**：`handle_diagnosis_run` → `diagnose_student` 读 `diagnosis.GRAPH_PATH`，`get_knowledge_points` 读 `api.KNOWLEDLEDGE_GRAPH_PATH`，两者是不同模块的独立常量。fixture 只 patch 一个会让另一条调用链落到真实文件，测试给出虚假信心。判断标准：跨层调用链上每个模块的路径常量都要 patch，或在 isolated_db fixture 内统一 patch 所有路径常量。
3. **死代码是 schema 与实现错位的信号**：`_evaluate_risk` 定义了 gray 评级，但 `diagnose_student` 只遍历有错题的知识点，gray 永不产出。这是 schema 承诺（四色评级）与实现（只评三级）的错位。修复时不要删 gray 定义，而要让实现兑现 schema 承诺（遍历全部图谱节点取并集评级）。判断标准：schema 字段的每个枚举值都应有对应的产出路径，否则是死代码。
4. **HaloRead 沿袭迁移必须 Grep 校验所有通用工程 skill，不能只改 rules/checklists**：`bug-reporting.md` 和 `dev-checklist.md` 改了 `output → content`，但 3 个通用工程 skill（dev-selfcheck/verification-before-completion/git-merge-guardian）仍写 `--output output`。迁移后必须 Grep `.trae/skills/` 全目录，区分豁免范围（deep-reading/ + prompts/）与必改范围（通用工程 skill）。
5. **三视角并行评审的 P0/P1 修复要分优先级，暂不修的记入沉淀**：本次 3 subagent 评审发现 3 P0 + 多个 P1 + 多个 P2。P0 全修（阻塞合并）；P1 按影响面挑关键修（架构 P1-3/P1-4 + 测试 P1-3/P1-6 + 规则 P1-1/P1-2）；P2 选修（P2-2 去重/P2-4 None 防御/P2-5 yellow 覆盖）；暂不修的（DiagnosisEngine 抽象层 / api.py 跨层调用 / 图谱路径硬编码 / 两阶段写入无事务 / README 更新 / LLM_MOCK 跨层分裂）记入沉淀作为 Phase 1.5/2 后续清理项。判断标准：P0 必修，P1 按影响面挑修，P2 选修，暂不修的必须有明确后续阶段承接。
6. **isolated_db fixture 应 patch 所有持久化输出目录，不只 DB 路径**：原 fixture 只 patch `store.DB_PATH` 和 `wrongbook.MD_BACKUP_DIR`，漏了 `exam.OUTPUT_DIR` 和 `voice.VOICE_OUTPUT_DIR`，导致 exam/voice 测试向真实 `data/exam_output` 写文件。判断标准：每新增一个 service 写入路径，isolated_db fixture 必须同步追加 patch，并用 `git status` 验证测试运行后 `data/` 无新文件。

### 可复用资产

- **Python 默认参数陷阱修复模板**：函数默认参数改 `db_path: Path | None = None` + 函数体内 `if db_path is None: db_path = DB_PATH`；模块级常量改为 `_get_xxx()` 函数内 `from .. import store; return store.DB_PATH.parent / "xxx"`。配套守护测试 `test_default_xxx_resolved_at_call_time`：monkeypatch 全局变量后不传参调用，断言使用新路径。
- **三视角并行评审 P0/P1 修复优先级矩阵**：P0 阻塞合并必修 / P1 按影响面挑修（架构影响 > 测试覆盖 > 规则合规）/ P2 选修（去重/防御/覆盖）/ 暂不修必须有后续阶段承接。可作为后续 Phase 评审修复的标准优先级矩阵。
- **isolated_db fixture 全持久化 patch 模板**：`store.DB_PATH` + `wrongbook.MD_BACKUP_DIR` + `exam.OUTPUT_DIR` + `voice.VOICE_OUTPUT_DIR` 四件套，每新增 service 写入路径同步追加。配套 `git status` 验证测试运行后 `data/` 无新文件。
- **graph_path fixture 跨层 patch 模板**：同时 patch `api.KNOWLEDLEDGE_GRAPH_PATH` 和 `diagnosis.GRAPH_PATH`，确保诊断链路与知识点列表查询都走临时图谱。配套断言 root_causes + recommendation_path，守护图谱遍历在 api 层的集成。
