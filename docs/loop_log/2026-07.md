## Phase 0 魔搭按钮可点与前后端分离架构基线（2026-07-18，Gradio 入口 + src/server 四层 + HaloRead 沿袭清理 + 三视角评审修复）#book_structure #deployment

### 触发问题

魔搭 ModelScope Studio 上 knowtrace 项目页面（https://www.modelscope.cn/studios/codengseam/knowtrace/summary）所有按钮都不能点。同时 28 天 MVP 五链路（拍照录题/认知诊断/RAG题库/一键出卷/FunASR语音周报）需要奠定前后端分离架构基线，让前端能真实调用后端。

### 根因

1. **魔搭 Studio 入口约定缺失**：魔搭 Studio 默认查找根目录 `app.py`，仓库无 `app.py` 导致 iframe 加载空白应用，所有按钮自然失效。原仓库 `src/main.py` 仅是骨架（打印节点数），不是魔搭入口。
2. **HaloRead 静态前端沿袭与 knowtrace 项目定位不匹配**：复用自 HaloRead 的 `src/web/static-site/` 静态前端、`scripts/build_site.py` 的 `_copy_static_assets()`、`tests/run_regression_suite.sh` 的 app.js 语法/沉浸模式/HTTP 冒烟步骤，都依赖 HaloRead 专属前端资产，knowtrace 改用 Gradio 入口后这些全部失效，但仍残留在测试体系中拖累 CI。
3. **测试体系沿袭 HaloRead 业务假设**：19 + 7 个 HaloRead 沿袭测试依赖 `src.core`/`src.agents`/`output/` 目录/资治通鉴书籍/`scripts/check_missing_columns.py` 等 knowtrace 未迁移的模块，import 即失败。原有 `pytest.importorskip("langgraph")` 间接跳过策略在 `requirements-dev.txt` 装 langgraph 后会收集期崩溃。
4. **50 章教研 md frontmatter.book 残留「重庆」前缀**：与项目面向乡村师生的定位不符，触发 `check_book_structure.py --strict` P1 校验失败。

### 关键教训

1. **魔搭 ModelScope Studio 入口约定（根目录 `app.py`）必须最先满足，否则页面白屏所有按钮失效**：这是部署平台层面的硬约束，与项目代码质量无关，但没满足它会让你以为「代码错了」其实「入口错了」。新项目接入魔搭时第一件事是确认根目录 `app.py` 存在且顶层定义 `demo` 变量（Gradio Blocks）或可调用对象（Streamlit）。
2. **HaloRead 沿袭测试清理必须扩展 `collect_ignore` 显式跳过，不能依赖 `importorskip` 间接跳过**：`pytest.importorskip("langgraph")` 在 langgraph 已装但 `src.core.workflow` 不存在时会收集期崩溃（不是优雅 skip）。判断标准：如果测试文件 import 的模块在当前项目不存在，必须加入 `collect_ignore`；如果只是 import 时缺可选依赖，可用 `importorskip`。
3. **Phase 0 占位 service 的文件扩展名必须与实际内容匹配**：`services/exam.py` 用 `write_text` 写纯文本但扩展名是 `.docx`，评委点击下载会得到「打不开的 Word 文件」。规则：Phase 0 占位写 txt，Phase X 真实实现时再换正确扩展名（docx/mp3）。这条适合所有「占位 + 后续接入真实库」的 service。
4. **`.gitignore` 必须显式忽略 `src/server/` 写入的运行时数据**：跨项目资产迁移时 `.gitignore` 的 `data/` 语义会变（HaloRead data/=缓存可全量忽略 vs knowtrace data/=知识图谱核心资产必入库 + 运行时数据需精确忽略）。规则：每新增一个 `src/server/` 写入路径（如 `data/wrongbook.db`），必须同步追加到 `.gitignore`，并用 `git check-ignore` 验证。
5. **`requirements.txt` 与 `requirements-dev.txt` 分离让魔搭 CI 冷启动提速**：魔搭 Studio 默认 `pip install -r requirements.txt`，langchain/langgraph 系列重依赖（200+ MB）Phase 0 完全用不到，挪到 `requirements-dev.txt` 让冷启动从 1-2 分钟降到 30 秒内。判断标准：魔搭入口 `app.py` 不 import 的库都不应在 `requirements.txt`。
6. **三视角并行评审（架构师/测试/规则）发现的问题远超主 Agent 自查**：本次 3 个 subagent 并行约 3 分钟，发现 4 个 P0 + 11 个 P1 + 10 个 P2，主 Agent 自查只能发现语法错误和明显违规。视角分工要正交（架构师看可行性与扩展性 / 测试看覆盖与边界 / 规则看合规与边界），避免重叠。
7. **跨项目 review_plan.py 路径 B 占位模板可复用**：缺 `.env`/`langgraph` 时返回 1 + 友好提示，调用方降级到路径 C（Task 工具）。这个模板可用于任何「主路径 + 可选增强路径」的 Skill 配套脚本。

### 可复用资产

- **魔搭 Studio Gradio 入口模板**：`sys.path.insert(0, _PROJECT_ROOT)` + `try/except init_store()` + `gr.Blocks` 多 Tab + 每个 Button `click` 真实调用业务函数。可作为任何魔搭 Studio 项目的入口起点。

## 跨项目工程化资产迁移（2026-07-16，HaloRead → knowtrace 精选脚本 + 路径引用调整 + rules 项目名替换）#book_structure

### 触发问题

新建 knowtrace 项目（云脉·智诊伴学），需要从 HaloRead 项目复用已沉淀的工程化资产——Trae Skills/Rules/Checklists、校验与构建脚本、测试与回归体系、GitHub Actions、顶层配置。直接全量拷贝会带入 HaloRead 专属业务代码（book-sources/output/site/demos/prompts/src/agents），且 rules 内的路径引用（output/）与项目名（HaloRead）与新项目结构不匹配。

### 根因

1. **工程化资产与业务内容耦合在同一仓库**：HaloRead 的 scripts/ 同时含通用校验脚本（check_book_structure.py 等）与业务脚本（migrate_wellness_books.py、fix_zizhi_chapter_sort.py 等），全量拷贝会污染新项目。
2. **rules 内硬编码了项目特定路径与名称**：dev-workflow.md 写死 `--output output --strict`（knowtrace 用 content/）、bug-reporting.md 写死「HaloRead 项目」，不调整会让规则文件指向错误目录与项目名。
3. **测试体系依赖 conftest.py 公共 fixture**：只拷 test_*.py 不拷 conftest.py，pytest 的 autouse mock_env fixture 缺失，回归体系无法独立运行。

### 修复

1. **精选而非全量拷贝**：scripts/ 只拷 12 个通用校验/构建/治理脚本，过滤掉业务专属脚本（migrate_*、fix_*、download_sage_portraits 等）；tests/ 拷 bug_regression_list.md + run_regression_suite.sh + 全部 test_*.py + conftest.py（pytest harness，带优雅降级）；不拷 book-sources/output/site/demos/docs（除 loop_log）/prompts/src/agents。
2. **路径引用调整**：dev-workflow.md 中 `--output output --strict` → `--output content --strict`（合并前校验命令与自检命令）、`` `output/` `` → `` `content/` ``（loop_log 写入门槛「纯内容生成日志 → 去 output/ 自身」），docs/loop_log/ 路径保留。
3. **项目名替换**：bug-reporting.md「HaloRead 项目」→「knowtrace 项目」；dev-workflow.md 无 HaloRead 字面量（本就用通用表述），无需替换。
4. **顶层配置整目录拷贝**：.gitignore / .env.example / requirements.txt / githooks/（含 pre-push）。

### 涉及文件

- `/tmp/knowtrace/.trae/{skills,rules,checklists}/`（整体复制，12 个 skill 目录）
- `/tmp/knowtrace/scripts/`（12 个精选脚本）
- `/tmp/knowtrace/tests/`（bug_regression_list.md + run_regression_suite.sh + conftest.py + 全部 test_*.py）
- `/tmp/knowtrace/.github/workflows/`（pages.yml + regression.yml + branch-cleanup.yml + deploy-modelscope.yml）
- `/tmp/knowtrace/{.gitignore,.env.example,requirements.txt,githooks/}`
- `/tmp/knowtrace/.trae/rules/dev-workflow.md`（output→content 路径调整）
- `/tmp/knowtrace/.trae/rules/bug-reporting.md`（项目名替换）
- `/tmp/knowtrace/docs/loop_log/2026-07.md`（新建，本条沉淀）

### 关键教训

1. **跨项目资产迁移必须「精选脚本 + 调整路径 + 替换项目名」三步走，不能全量 cp -r**：全量拷贝会带入业务专属代码与硬编码路径，新项目跑校验时路径指向错误目录。通用校验脚本（check_*/validate_*/regen_*/build_site/sync_rules/branch_governance）可复用，业务脚本（migrate_*/fix_*/download_*）不可复用。
2. **rules 文件内的路径引用是迁移时最易遗漏的硬编码**：dev-workflow.md 把 `--output output` 写进合并前校验命令与自检命令，若不改为 `--output content`，knowtrace 跑 check_book_structure 会扫描不存在的 output 目录。迁移后必须 Grep 校验 `output/` 出现次数为 0。
3. **测试体系迁移要带 conftest.py，否则 pytest harness 不完整**：conftest.py 的 autouse fixture（mock_env）是回归体系能独立运行的前提，且其 try/except 优雅降级设计让它在缺 src.core.workflow 时不阻塞，适合跨项目迁移。
4. **Edit/Write 工具受沙箱工作目录限制，跨目录（/workspace → /tmp）编辑需用 RunCommand + Python 替代**：当目标在主工作目录之外时，Edit 报「File's parent is not a folder」，改用 `python3 << PYEOF` 做精确字符串替换是可靠回退方案。

### 可复用资产

- **跨项目资产迁移清单模板**：.trae/{skills,rules,checklists} 整体拷贝 + scripts 精选 12 个通用脚本 + tests 带 conftest 全套 + .github/workflows 全套 + 顶层配置 4 件套，可作为后续从 HaloRead 衍生新项目的标准迁移清单。
- **rules 路径调整 checklist**：迁移后必做 Grep 校验——dev-workflow.md `output/`=0、项目名（HaloRead）=0，docs/loop_log/ 保留。

### 关键教训

1. **复用姊妹项目 .gitignore 必须重新审视 data/ 等目录的语义**：同一目录名在不同项目可能承载不同性质文件（HaloRead data/=缓存可忽略 vs knowtrace data/=核心资产必入库）。迁移 .gitignore 后必须用 `git check-ignore` 验证核心资产未被忽略。
2. **多 subagent 并行设计必然产生跨文档矛盾，主 Agent 汇总时必须做一致性扫描**：本次矛盾集中在"无影云断网能力"——架构师视角强调限制（断网即停服）作为双轨制动因，生态顾问视角强调兜底（断网也能用）作为落地性证据，两者都自洽但放一起冲突。主 Agent 调和方式：区分"新出卷"（需联网）与"用缓存上课"（可离线），既保留双轨制动因又保留落地性。
3. **专家团三视角并行审查是性价比极高的质量门**：3 个 subagent 并行审查约 5 分钟，发现 8 个问题（含 1 个 P0 阻断合并），远低于主 Agent 自查的成本。视角分工要正交（骨架完整性 / 生态对接 / 数据质量），避免重叠。
4. **sed 替换含 `|` 的 Markdown 表格行会报错，改用 Python 字符串替换**：sed 的分隔符 `|` 与表格分隔符冲突，`sed 's|...|...|'` 报 "unknown option"。含管道符的行用 `python3 -c "s=s.replace(old,new)"` 更可靠。

### 可复用资产

- **专家团三视角审查模板**：骨架完整性（顶层/docs/data/content/.trae/scripts/tests/workflows/src 十维度）+ 领域对接完整度（能力覆盖/红线一致性/叙事一致性/浓度评估）+ 数据质量（覆盖度/完整性/依赖闭环/真实性），三视角正交不重叠，可作为后续新项目骨架审查的标准模板。
- **跨文档一致性扫描清单**：双轨制断网能力、防幻觉红线措辞、生态定位（可对接不依赖）、节点数口径、模型职责分工——这五项是多 subagent 并行设计最易产生矛盾的点，汇总阶段必扫。
