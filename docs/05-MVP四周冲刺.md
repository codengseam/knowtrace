# 05 · MVP 四周冲刺路线

> 单人参赛，28 天（2026-07-16 至 2026-08-13）。时间是最大敌人，"做减法"和"抓核心"是唯一出路。

## 一、冲刺总原则

1. **先跑通主干，再丰富枝叶**——Loop Agent 敏捷迭代，先有可演示的"伪闭环"再补全。
2. **测试开发降维打击**——优势是兜底与鲁棒性（try-except / 重试 / 默认推荐库 / 边界 case 构造），不与算法工程师比模型调参。
3. **每日可演示**——每周末必须有可跑的 Demo 雏形，不允许"埋头三天无产出"。
4. **不追求全，追求准**——知识图谱宁少勿错，MVP 只做七年级上册 50 节点。

## 二、四周分解

### Week 1 · 基建与核心链路（Day 1-7）

**目标**：跑通"伪闭环"——上传错题 → 识别 → 诊断 → 展示溯源结果。

| Day | 任务 | 交付物 |
|---|---|---|
| 1-2 | 百炼平台搭建 RAG 应用，导入 500 道初中数学真题（重点一元一次方程相关） | RAG 应用 + 检索准确率报告 |
| 3-4 | 编写核心 Prompt（[diagnose_prompt.md](../data/prompts/diagnose_prompt.md)），调试 Qwen3-Max 错因分析 JSON 输出稳定性 | Prompt 模板 + 评测样本 |
| 5-7 | Gradio/Streamlit 极简 Web UI：上传错题图 → Qwen3-Omni 识别 → Qwen3-Max 诊断 → 展示溯源 | **可演示 Demo 雏形** |

**Week 1 验收**：能在本地跑通一道"因式分解错题"的完整诊断链路。

### Week 2 · 教师端杀手锏 + 图谱可视化（Day 8-14）

**目标**：教师端"一键出卷"交付物 + 学生端认知诊断树。

| Day | 任务 | 交付物 |
|---|---|---|
| 8-10 | 一键出卷：RAG 检索真题 → Qwen3-Max 排版生成解析 → python-docx 生成 A4 Word | 一键出卷功能 + 示例 Word |
| 11-12 | 前端 ECharts 或 AntV G6 渲染认知诊断树（红/黄/绿/灰四色节点） | 知识树可视化页面 |
| 13-14 | 整合学生端 Web UI，"拍照→诊断→看图→做题"链路丝滑 | 学生端完整 Demo |

**Week 2 验收**：教师输入"全班 25 人错在因式分解"，3 秒生成 A4 Word 试卷；学生看到带四色高亮的认知诊断树。

### Week 3 · 场景补齐 + 生态集成（Day 15-21）

**目标**：微信小程序（家场景）+ FunASR 语音周报 + 疯狂边界测试。

| Day | 任务 | 交付物 |
|---|---|---|
| 15-17 | 微信小程序或 H5：错题本列表 + 离线图片缓存（<50MB） | 小程序雏形 |
| 18-19 | FunASR 接入：学情周报文本 → 方言/普通话语音播报（60 秒内） | 语音周报功能 |
| 20-21 | 边界测试：模糊图片 / 乱写草稿 / 断网重连 / 公式渲染崩溃兜底 | Bug 修复 + 回归用例 |

**Week 3 验收**：FunASR 能生成"小明爷爷您好，小明这周数学方程部分进步很大..."的方言语音；断网重连不崩溃。

### Week 4 · PPT + 演示视频 + 提交（Day 22-28）

**目标**：商业计划书 + 3 分钟演示视频 + 提交。

| Day | 任务 | 交付物 |
|---|---|---|
| 22-24 | 撰写商业计划书 / PPT：痛点 → 阿里生态缺位 → 方案（双轨制+混合架构）→ 百炼赋能 → 落地 | PPT 初稿 |
| 25-26 | 录制 3 分钟演示视频，含两个高光时刻（学生顿悟 + 教师一键出卷） | 演示视频 |
| 27-28 | 提交材料 + 答辩 Q&A 准备（见 [06-演示剧本与Checklist](06-演示剧本与Checklist.md)） | 提交包 |

**Week 4 验收**：材料齐备，对照 [提交 Checklist](06-演示剧本与Checklist.md) 逐项打勾。

## 三、每日 Loop 节奏

```
早晨（10 分钟）：更新 todo，确认今日核心目标（一句话）
白天：执行，每完成一步短汇报
睡前（5 分钟）：跑校验（check_book_structure + pytest），记 loop_log
每周末：Demo 演练，发现翻车点立即修
```

## 四、风险与应对

| 风险 | 概率 | 应对 |
|---|---|---|
| Qwen3-Max 输出非法 JSON | 中 | system prompt 锁 schema + 二次提示修正 + try-except 兜底 |
| RAG 检索准确率不足 | 中 | 调 Top-K、加 metadata 过滤（年级+难度）、人工标注 50 道金标准 |
| 小程序审核超时 | 低 | 降级为 H5，不阻塞主线 |
| 演示视频翻车 | 中 | 预构造 3 个"完美错题案例"（因式分解错、求根公式错、配方法错） |
| 知识图谱依赖链不准 | 中 | 试点学校老师人工审核核心链路，宁少勿错 |

## 五、不做清单（守住减法）

- ❌ 不做物理 / 化学（MVP 只数学单科）
- ❌ 不做八年级以下（MVP 只七年级上册）
- ❌ 不做重度 Neo4j 图谱（用轻量 JSON）
- ❌ 不做端侧大模型（无影云无端侧算力）
- ❌ 不做夸克 / 钉钉官方 API 对接（比赛阶段拿不到）
- ❌ 不做复杂 BI 看板（老师要的是卷子不是看板）
- ❌ 不让 LLM 凭空生成数学题（防幻觉红线）

## 六、LoopAgent 沉淀纪律

每次开发完成后做沉淀复盘：

1. 本次改动是否暴露新共性问题？
2. 是否需要更新 `.trae/rules/` 或 `scripts/`？
3. 是否需要更新 `.trae/checklists/dev-checklist.md`？
4. 是否需要在 `docs/loop_log/2026-07.md` 追加一条开发沉淀记录？

详细沉淀规范见 `.trae/rules/dev-workflow.md` 第五步。
